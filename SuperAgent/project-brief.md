# RDPProxy v2 — Полное описание проекта

## 1) Контекст и цель

`rdpproxy v2` — это полная переработка `v1` из монолита в микросервисную архитектуру на Docker.
Проект решает задачу безопасной публикации RDP-доступа через единую внешнюю точку входа, сохраняя
весь бизнес-функционал v1 и добавляя масштабируемость, отказоустойчивость и расширяемость.

Базовый принцип:
- пользователь входит в веб-портал через LDAP/AD;
- получает список доступных RDP-целей по группам;
- скачивает `.rdp` с одноразовым токеном;
- RDP-сессия проходит через relay, где выполняется TLS/CredSSP/MCS-совместимость.

## 2) Что уже реализовано в v2

### 2.1 Функционал, перенесенный из v1

- LDAP/AD-аутентификация пользователей.
- Генерация `.rdp` файлов с параметрами шаблона.
- SessionStore в Redis (включая хранение пароля в шифрованном виде).
- CredSSP/NTLM-аутентификация на backend RDP-сервер.
- MCS patching для совместимости negotiation SSL/HYBRID.
- Учёт активных/исторических подключений.
- Админ-функции управления серверами, шаблонами, группами, пользователями.

### 2.2 Новые требования, добавленные в v2

- Микросервисная декомпозиция: `portal`, `admin`, `rdp-relay`, `metrics`, `haproxy`, `postgres`, `redis`.
- Всё в Docker (включая PostgreSQL и Redis), без host-mode сервисов.
- Docker bridge-сеть вместо `network_mode: host`.
- Единый внешний порт `443` для HTTPS+RDP через L4 mux в HAProxy.
- Proxy Protocol v2 для RDP (получение реального IP на relay).
- `X-Forwarded-For` / `X-Real-IP` для HTTP-пути (portal/admin).
- Админка на `9090` с LAN-ограничением (привязка к `${LAN_IP}` + nftables).
- Подготовка к multi-node: модель `ClusterNode`, heartbeat через metrics, мониторинг нод.
- Плагинная архитектура в relay (без изменения ядра при добавлении новых функций).
- Бэкап БД через `pg_dump` скрипт.

## 3) Бизнес-требования и текущие решения

### 3.1 Единая точка входа
- Публичный вход только через HAProxy: `:443`.
- HTTP/HTTPS и RDP делятся на L4-уровне по сигнатуре трафика.

### 3.2 Безопасность
- Шифрование чувствительных данных в Redis: AES-256-GCM.
- Пароли локальных админов: Argon2.
- CSRF защита для web-форм.
- Rate limiting и временные lockout для логина.
- Ограничение админки по LAN.

### 3.3 Масштабирование и HA
- Старт с single-node (`docker compose up`) полностью поддержан.
- Архитектура готова к multi-node (горизонтальное масштабирование).
- При падении ноды активные TCP/RDP сессии на ней обрываются; новые подключения уходят на живые ноды.
- Репликация БД и Redis заложена как отдельные этапы эволюции (single -> replica/sentinel -> cluster-grade).

### 3.4 Расширяемость (Open/Closed)
- Новые RDP-фичи добавляются через плагины relay.
- Новые API-фичи добавляются отдельными router-модулями, без переписывания ядра.
- Новые таблицы добавляются через отдельные модели + Alembic-миграции.

## 4) Состав системы (в прод-модели)

- `haproxy` — внешний ingress, L4 mux + TLS termination + admin proxy.
- `portal` — пользовательский веб-портал (логин, список серверов, `.rdp`).
- `admin` — админ-панель (CRUD, аудит, мониторинг, кластерный обзор).
- `rdp-relay` — TCP relay для RDP (PPv2, X.224, TLS, CredSSP, plugins).
- `metrics` — сбор метрик и heartbeat ноды.
- `postgres` — основное персистентное хранилище.
- `redis` — сессии, токены, трекеры, rate-limit, быстрые heartbeat.

## 5) Ключевые сценарии

### 5.1 Пользовательский вход
1. Пользователь открывает портал по `https://<host>:443`.
2. Проходит LDAP-аутентификацию.
3. Портал вычисляет доступные сервера по группам.
4. Пользователь скачивает `.rdp` с токеном.
5. RDP-клиент подключается на `443`, HAProxy отправляет поток в relay.
6. Relay валидирует токен, делает CredSSP к backend, запускает двунаправленный прокси.

### 5.2 Админский контур
1. Админ входит через `http://<LAN_IP>:9090/admin/login`.
2. Управляет серверами, шаблонами, настройками, админами.
3. Смотрит активные сессии и историю.
4. Контролирует состояние нод/сервисов по heartbeat/статистике.

## 6) Нефункциональные требования

- Чистый и читаемый код с type hints.
- Минимизация связности между сервисами.
- Отказоустойчивость на уровне контейнеров и health-check.
- Быстрый онбординг новых AI-агентов за счет структурированной документации.

## 7) Операционные ограничения и принципы

- Не использовать host services для PostgreSQL/Redis в целевой модели.
- Не использовать `network_mode: host` для сервисов приложения.
- Порты наружу публикует только HAProxy.
- Все секреты хранятся вне git (`.env`, `config.yaml`).
